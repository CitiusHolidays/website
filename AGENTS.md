## TypeScript and Effect Migration Guardrails

- Migrate source files to TypeScript broadly, but keep plain TypeScript as the default implementation style.
- Use Effect only when a module has at least two real orchestration pressures: external I/O, retry or throttle behavior, concurrency control, typed recoverable errors, rollback or cleanup, and test-time dependency substitution.
- Do not use Effect for simple async functions, simple React component state, straightforward Convex validators, or ordinary Convex query/mutation business state transitions.
- Start Effect usage in backend/API orchestration seams such as payment routes, notification email delivery, spreadsheet imports, migrations, and other failure-prone integration edges before considering it in portal UI.
- Copy local Effect examples from `src/lib/effectAdoption.ts`; do not invent ad hoc Effect style.
- Use `bun run typecheck` for migrated app/source TypeScript. Keep Convex's own `convex/tsconfig.json` and `bunx convex codegen` flow separate.
- Finish in-flight TS/Effect migration slices with tests before unrelated greenfield work (Guardrailed Completion), unless the user explicitly pivots; office-hours may still capture separate feature direction.
